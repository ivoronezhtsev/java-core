# Состояние гонки (Race condition) #
Состояние гонки - ошибка проектирования, в рез-те которой рез-т выполнения программы непредсказуем и зависит от порядка 
выполнения потоков
### Причины ###
* __Общая область памяти (Shared State)__<br>
Потоки должны иметь доступ к одному и тому же ресурсу (переменной, файлу, объекту в памяти). 
Если у каждого потока свои данные, конфликтовать не с чем.
* __Изменяемость данных (Mutability)__<br>
Хотя бы один поток должен выполнять операцию записи. 
Если все потоки только читают данные (immutable state), состояние гонки невозможно, так как значение ресурса не меняется непредсказуемо.
* __Отсутствие синхронизации (Lack of Coordination)__
Когда нет механизмов (мьютексов, семафоров, атомарных операций), гарантирующих очередность или неделимость (атомарность) действий. Без этого порядок выполнения инструкций процессором или планировщиком ОС становится случайным, что и приводит к ошибкам.

### Классический пример: ###
Два потока одновременно делают x = x + 1.<br>
Оба читают x = 10.<br>
Оба прибавляют 1 (получают 11).<br>
Оба записывают 11.<br>
Итог: вместо 12 мы получили 11, потому что из-за отсутствия синхронизации один инкремент «потерялся».

```java
public class Demo {
    
    static int x = 0;

    public static void main(String[] args) throws Exception {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 100_000; i++) x++;
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 100_000; i++) x++;
        });
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println(x);
    }
    
}
```
Ожидали получить 200_000<br>
Здесь, как правило, x в рез-те будет меньше 200_000, но иногда может быть равен 200_000 (т.е. непредсказуем рез-т)<br>
Решается заменой int на AtomicInteger или synchronized блока
